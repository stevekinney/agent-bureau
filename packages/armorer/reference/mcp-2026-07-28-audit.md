# MCP surface audit: `createMCP` / `fromMcpTools` vs the 2026-07-28 draft revision

**Scope:** `packages/armorer/src/integrations/mcp/index.ts` (the entire MCP integration surface — `createMCP`, `toMcpTools`, `fromMcpTools`, and their supporting helpers). No other package in the monorepo imports `armorer/mcp` or the MCP SDK directly (`grep -rl "fromMcpTools\|createMCP\|armorer/mcp" packages` returns only armorer's own source/tests), so this file is the complete audit surface.

**Date of audit:** 2026-07-09.

**Spec baseline:** [modelcontextprotocol.io/specification/draft](https://modelcontextprotocol.io/specification/draft) — the **draft** revision, dated **2026-07-28**, which at time of writing has not yet been promoted to "Current" (the current published revision is still 2025-11-25). All spec claims below were fetched directly from `modelcontextprotocol.io` on the audit date and are cited by page. Anything not directly quoted from a fetched page is marked **inferred**.

**SDK baseline:** `@modelcontextprotocol/sdk@1.29.0` (resolved version per `bun.lock`, installed under `packages/armorer/node_modules/@modelcontextprotocol/sdk`; peer dependency range in `packages/armorer/package.json` is `^1.26.0`). This is the newest version published to npm as of the audit date (`npm view @modelcontextprotocol/sdk versions` tops out at `1.29.0`).

## Headline finding

**The installed MCP TypeScript SDK does not implement the 2026-07-28 draft at all.** `packages/armorer/node_modules/@modelcontextprotocol/sdk/dist/esm/types.js` line 2 hardcodes:

```js
export const LATEST_PROTOCOL_VERSION = '2025-11-25';
export const DEFAULT_NEGOTIATED_PROTOCOL_VERSION = '2025-03-26';
export const SUPPORTED_PROTOCOL_VERSIONS = [
  LATEST_PROTOCOL_VERSION,
  '2025-06-18',
  '2025-03-26',
  '2024-11-05',
  '2024-10-07',
];
```

A repo-wide search for the draft's new wire vocabulary against the installed SDK returns **zero matches**:

```
grep -rEn "ttlMs|cacheScope|resultType|inputRequests|inputResponses|requestState|InputRequiredResult|Mcp-Method|Mcp-Name|traceparent|server/discover|subscriptions/listen" \
  node_modules/@modelcontextprotocol/sdk/dist/esm/server/*.js node_modules/@modelcontextprotocol/sdk/dist/esm/types.js
# (no output)
```

Since `createMCP` (`src/integrations/mcp/index.ts:99-153`) constructs an SDK `McpServer` and delegates all wire-protocol behavior to it (session negotiation, header handling, list-caching hints, MRTR result shapes), and `fromMcpTools` (`src/integrations/mcp/index.ts:163-186`) is transport-agnostic and only shapes whatever `CallToolResult` its `callTool` callback returns, **every gap below traces back to the SDK dependency, not to logic armorer owns**. Armorer's own ~570 lines of glue code contain no protocol-version assumptions that would block adopting a 2026-07-28-capable SDK release when one ships — the one exception (JSON Schema `$ref`/2020-12 handling in our own converter) is called out separately below.

## Criterion-by-criterion audit

### 1. Stateless transport (no `initialize`/session-id dependence)

**Spec (draft, fetched from `/specification/draft/basic/index`, section "Statelessness"):**

> "The Model Context Protocol (MCP) is a **stateless protocol**: all the information needed to process a request is contained in the request itself. … Servers **MUST NOT** rely on prior requests over the same connection to establish context (e.g., capabilities, protocol version, client identity)."

And from `/specification/draft/changelog`, major change #2: "Make MCP stateless: remove the `initialize`/`notifications/initialized` handshake."

**Our code:** `createMCP` (`src/integrations/mcp/index.ts:99-153`) never references `initialize`, `sessionId`, or `Mcp-Session-Id` — it has no involvement in connection lifecycle at all; it only registers tools onto whatever `McpServer` instance it creates and returns that server for the caller to `.connect()` to a transport. `grep -En "sessionId|initialize|Mcp-Session" src/integrations/mcp/index.ts` returns no matches.

**Verdict: Supported by omission, gap in the delegate.** Armorer's server-construction code has no hard dependency on the stateful `initialize` handshake or `Mcp-Session-Id` — it would work unmodified against a stateless transport. But the _actual_ statefulness behavior is entirely the SDK transport's call: `packages/armorer/node_modules/@modelcontextprotocol/sdk/dist/esm/server/webStandardStreamableHttp.js:63` (`this.sessionIdGenerator = options.sessionIdGenerator`) shows the SDK's `WebStandardStreamableHTTPServerTransport` is **stateless by default** today (no `sessionIdGenerator` passed → `undefined` → no `Mcp-Session-Id` issued, per the class doc comment at lines 17-27 of that file) — but the SDK still implements the _2025_ `initialize`/`notifications/initialized` handshake as a first-class concept (`SUPPORTED_PROTOCOL_VERSIONS` above), which the 2026-07-28 draft removes outright. Our `documentation/mcp.md:34-38, 56-68` examples construct `StreamableHTTPServerTransport`/`McpServer` with no session options, i.e. we already document the stateless-friendly path, but we do not (and, given the SDK, cannot) implement the draft's `_meta`-carried-per-request identity model — see §2.

### 2. `_meta` client metadata (`io.modelcontextprotocol/protocolVersion`, `clientInfo`, `clientCapabilities`) and `InputRequiredResult`/`requestState` retry pattern

**Spec (draft, fetched from `/specification/draft/basic/index`, "`_meta`" section):**

> "Every client request **MUST** include the following `io.modelcontextprotocol/*` fields in `_meta`. … A request missing any required field is malformed; the server **MUST** reject it with JSON-RPC error code `-32602`."

Table of required keys: `io.modelcontextprotocol/protocolVersion`, `io.modelcontextprotocol/clientInfo`, `io.modelcontextprotocol/clientCapabilities` (all required), `io.modelcontextprotocol/logLevel` (optional).

**Spec (draft, fetched from `/specification/draft/server/tools`, "Input Required Tool Results"):**

> "Servers **MAY** respond to `tools/call` with an `InputRequiredResult` … When retrying the request with input responses, clients include `inputResponses` and, if provided by the server, `requestState` in the request parameters … Note that the JSON-RPC `id` **MUST** be different between the initial request and the retry."

**Our code:** The SDK's `RequestHandlerExtra` type (`node_modules/@modelcontextprotocol/sdk/dist/esm/shared/protocol.d.ts:173-197`) already carries an `_meta?: RequestMeta` field on every request handler invocation — but `toMcpToolDefinition`'s handler (`src/integrations/mcp/index.ts:209-237`) only destructures `extra?.requestId` (line 213) and `extra?.signal` (line 224); it never reads `extra?._meta`. There is no code path anywhere in `src/integrations/mcp/index.ts` that reads or writes `io.modelcontextprotocol/protocolVersion`, `clientInfo`, `clientCapabilities`, `resultType`, `inputRequests`, `inputResponses`, or `requestState` (`grep -En` for all of those across the file: zero matches). `FromMCPToolsOptions['callTool']` (`src/integrations/mcp/index.ts:78-84`) has signature `(request: { name: string; arguments?: Record<string, unknown> }) => Promise<CallToolResult>` — no `_meta`, no way to pass or receive `inputResponses`/`requestState` on a retry.

**Verdict: Gap — both upstream and in our own glue.** The SDK does not implement the `resultType`/`InputRequiredResult`/MRTR pattern at all (confirmed absent from `types.js`), so there is nothing yet for armorer to plumb through even if it wanted to. But independent of the SDK, `FromMCPToolsOptions.callTool`'s narrow signature (name + arguments only) would need to grow `_meta` and `requestState` parameters to support MRTR once the SDK exposes it — this is a design gap in our own type, not solely inherited.

Note the distinction from tool-_level_ `_meta`, which armorer already handles today: `toMcpToolDefinition` sets `mcpTool._meta` from a tool's `metadata`/`toolConfigurationFromMetadata` output (`src/integrations/mcp/index.ts:249-251`). That's per-_tool_ metadata attached to a `Tool` definition at registration time, unrelated to the per-_request_ client-identity `_meta` fields (`protocolVersion`/`clientInfo`/`clientCapabilities`) this criterion is about. Those request-level fields are the gap; tool-level `_meta` is not.

### 3. `Mcp-Method` / `Mcp-Name` headers

**Spec (draft, fetched from `/specification/draft/basic/transports/streamable-http`, "Standard Request Headers"):**

> | Header Name | Source Field | Required For |
> | `Mcp-Method` | `method` | All requests |
> | `Mcp-Name` | `params.name` or `params.uri` | `tools/call`, `resources/read`, `prompts/get` requests |
> "These headers are **REQUIRED** for compliance."

**Our code:** `createMCP`/`toMcpTools`/`fromMcpTools` are entirely transport-agnostic — they never touch an HTTP `Request`/`Response` or header map (`grep -En "Mcp-Method|Mcp-Name" src/integrations/mcp/index.ts documentation/mcp.md`: zero matches). Header construction is delegated wholesale to whichever SDK transport class the caller instantiates (`documentation/mcp.md:60-68` shows callers building `StreamableHTTPServerTransport` themselves and wiring it to raw HTTP).

**Verdict: Gap, entirely upstream.** Confirmed by the same SDK grep in the headline finding — `Mcp-Method`/`Mcp-Name` do not appear anywhere in the installed SDK's server transport code. This is not something armorer's transport-agnostic design can address by itself; it would need the SDK's `StreamableHTTPServerTransport`/`WebStandardStreamableHTTPServerTransport` to emit and validate these headers. Because armorer never constructs the transport (only the `McpServer`), no code change on our side is blocked on this — it is a pure SDK-version wait.

### 4. `tools/list` TTL caching (`ttlMs`, `cacheScope`)

**Spec (draft, fetched from `/specification/draft/server/utilities/caching`):**

> "Servers MUST include caching hints on results with `resultType: 'complete'` returned by the following operations: … `tools/list` … Servers **MUST** provide a `ttlMs` value that is `>= 0`." Response example shows `"ttlMs": 300000, "cacheScope": "public"` alongside the tool list.

**Our code:** `createMCP`'s tool registration loop (`src/integrations/mcp/index.ts:145-147`) calls `registerTool` (line 109-123), which calls the SDK's `server.registerTool(toolName, config, tool.handler)` (line 116-120) — the `tools/list` response itself is assembled entirely inside the SDK (`node_modules/@modelcontextprotocol/sdk/dist/esm/server/mcp.js:67-95`, the `ListToolsRequestSchema` handler), which returns `{ tools: [...] }` with no `ttlMs`/`cacheScope` fields at all. There is no hook in `CreateMCPOptions` (`src/integrations/mcp/index.ts:86-92`) or `MCPToolConfiguration` (`src/integrations/mcp/index.ts:33-40`) for a caller to supply cache hints even if the SDK grew support for them.

**Verdict: Gap, both upstream and in our own option surface.** The SDK doesn't emit `ttlMs`/`cacheScope` (confirmed absent), and even if it did, `createMCP` has no configuration knob to set them — the tools/list response shape is entirely SDK-internal and not something `toMcpToolDefinition`/`toMcpRegisteredToolConfiguration` (lines 188-288) currently surface a way to influence.

### 5. W3C trace-context propagation

**Spec (draft, fetched from `/specification/draft/basic/index`, "OpenTelemetry trace context" subsection of `_meta`):**

> "As an exception to the prefix requirement above, the keys `traceparent`, `tracestate`, and `baggage` are reserved for OpenTelemetry trace context propagation. When present, their values MUST follow [W3C Trace Context] and [W3C Baggage] formats respectively."

**Our code:** No occurrence of `traceparent`, `tracestate`, or `baggage` anywhere in `src/integrations/mcp/index.ts` (confirmed by grep in the headline finding). Two concrete spots where this matters:

- **Inbound (server side):** the SDK's `RequestHandlerExtra._meta` (which would carry `traceparent` if a client sent it) is available to `toMcpToolDefinition`'s handler (`src/integrations/mcp/index.ts:209`) via the `extra` parameter, but the handler discards everything except `requestId`/`signal` (lines 213, 224) before calling `options.executeTool`/`tool.executeWith` (lines 214-227) — trace context from an incoming MCP call is never forwarded into the underlying Toolbox `execute`/`executeWith` call, so a tool's own OpenTelemetry spans (armorer already depends on `@opentelemetry/api` per `package.json`) cannot be parented to the caller's trace.
- **Outbound (client side):** `executeMcpTool` (`src/integrations/mcp/index.ts:330-345`) calls `options.callTool({ name, arguments })` with no `_meta`, so `fromMcpTools` has no way to inject `traceparent`/`tracestate`/`baggage` onto an outbound `tools/call` even if the caller-supplied `callTool` implementation wanted to honor it — the `FromMCPToolsOptions['callTool']` type (lines 78-84) simply doesn't carry the field.

**Verdict: Gap — this one is entirely ours to fix, not blocked on the SDK.** `armorer` already has an OpenTelemetry peer dependency and `RequestHandlerExtra._meta` is present in the _currently installed_ SDK version (it's a general-purpose `_meta` bag, not draft-specific — `node_modules/@modelcontextprotocol/sdk/dist/esm/shared/protocol.d.ts:189`), so nothing prevents wiring `extra?._meta` through to tool execution context today, independent of whether the draft spec's stateless/MRTR changes land. This is the strongest candidate for near-term work of everything audited here.

### 6. JSON Schema 2020-12 in tool schemas

**Spec (draft, fetched from `/specification/draft/basic/index`, "JSON Schema Usage"):**

> "1. **Default dialect**: When a schema does not include a `$schema` field, it defaults to JSON Schema 2020-12 … 3. **Supported dialects**: Implementations MUST support at least 2020-12 … Clients and servers **MUST** support JSON Schema 2020-12 for schemas without an explicit `$schema` field."
> And, on `$ref`: "JSON Schema 2020-12 permits `$ref` to point at an absolute URI. Implementations **MUST NOT** automatically dereference `$ref` values that resolve to a network URI." (`/specification/draft/basic/index`, "`$ref` Resolution").
> Also (fetched from `/specification/draft/server/tools`, "Tool" data type): "`inputSchema`: JSON Schema defining expected parameters … Defaults to 2020-12 if no `$schema` field is present."

**Our code, outbound direction (Toolbox tool → MCP tool schema):** `toMcpToolDefinition` (`src/integrations/mcp/index.ts:188-254`) passes the tool's Zod `input` schema straight through to the SDK's `server.registerTool` (line 116-120) as `inputSchema`. The SDK's own Zod→JSON-Schema conversion (`node_modules/@modelcontextprotocol/sdk/dist/esm/server/mcp.js:76-81`, calling `toJsonSchemaCompat(obj, { strictUnions: true, pipeStrategy: 'input' })` with **no `target` option**) falls back to `mapMiniTarget(undefined)` → `'draft-7'` (`node_modules/@modelcontextprotocol/sdk/dist/esm/server/zod-json-schema-compat.js:10-16`) for Zod v4 schemas, and the vendored `zodToJsonSchema` (v3 branch, same file lines 24-27) similarly defaults to draft-07-shaped output — **and neither branch emits a `$schema` field**, so a spec-conformant client receiving our tool's `inputSchema` would (per the "defaults to 2020-12 if no `$schema` field" rule quoted above) interpret a draft-07-shaped schema _as if it were 2020-12_. In practice the two dialects agree closely for the simple `type`/`properties`/`required`/`enum` shapes armorer's Zod schemas typically produce, so this is unlikely to cause validation divergence today, but it is a spec-conformance gap, not merely a style choice.

**Our code, inbound direction (MCP tool schema → executable Toolbox tool):** `fromMcpTools` (`src/integrations/mcp/index.ts:163-186`) calls `resolveMcpSchema(mcpTool.inputSchema)` (line 168), which (`src/integrations/mcp/index.ts:516-524`) delegates non-Zod schemas to `jsonSchemaToZod` (`src/json-schema-to-zod.ts`). That converter's doc comment (`src/json-schema-to-zod.ts:6-11`) enumerates exactly what it handles: `type`, `properties`, `required`, `items`, `enum`, `const`, `anyOf`, `oneOf`, `allOf`, `additionalProperties`, `nullable`, `description`, `default`. It has **no handling for `$ref`, `$defs`, `$schema`, `prefixItems`, or any 2020-12-specific keyword** — a schema using `$ref` (which the draft spec explicitly discusses resolution rules for, implying servers are expected to use it) would silently fall through to `undefined` for that subschema wherever `jsonSchemaToZod` recurses into an unrecognized shape (see the type/shape dispatch starting at `src/json-schema-to-zod.ts:47`), and `fromMcpTools` falls back to `z.object({}).passthrough()` (line 168) for the whole tool input if the top-level schema itself is unresolvable — a partially-`$ref`-based schema degrades silently to accepting arbitrary input rather than erroring.

**Verdict: Gap, entirely ours, independent of the SDK version.** This is not blocked on an SDK upgrade — `jsonSchemaToZod` is armorer's own converter (`src/json-schema-to-zod.ts`), and neither direction (outbound generation nor inbound consumption) is 2020-12-conformant today. Outbound is a minor/practical gap (no `$schema` field, effectively draft-07 shape); inbound is more serious (no `$ref` support at all, silent-permissive fallback on unresolvable schemas).

Five of the six criteria are gaps today; the sixth (statelessness) is partially supported — our own code is already compatible, but full conformance still depends on the transport. The distinguishing question for prioritization is **who can close it**:

- SDK-blocked (nothing armorer can do until `@modelcontextprotocol/sdk` ships 2026-07-28 support): full stateless-transport conformance (removal of `initialize`), `_meta`-carried protocol identity, MRTR/`InputRequiredResult`, `Mcp-Method`/`Mcp-Name` headers, `tools/list` caching hints.
- Ours to fix independent of SDK version: trace-context propagation (inbound `extra._meta` → tool execution context, outbound `callTool` `_meta` parameter), and JSON Schema 2020-12 conformance in `jsonSchemaToZod` (`$ref`/`$defs` support, stricter failure mode instead of silent `passthrough()`).

## Already-scheduled follow-ups (not re-filed here)

Per the roadmap, the following are already tracked as separate items and are referenced rather than duplicated:

- Sampling/roots removal (deprecated in the draft per SEP-2577) — **AB-31**
- Elicitation changes (removal of `notifications/elicitation/complete` and `elicitationId`, correlation via `requestState` instead) — **AB-32**
- OAuth changes (`iss` validation per RFC 9207, DCR deprecation in favor of Client ID Metadata Documents, `application_type` requirement) — **AB-33**
- Tasks extension redesign (`io.modelcontextprotocol/tasks`, `tasks/get`/`tasks/update` replacing `tasks/result`) — **AB-34**

## Unscheduled follow-ups (found during this audit, not covered by AB-31..34)

1. **Trace-context propagation is entirely unimplemented in `armorer/mcp`, and is not blocked on the SDK.** `toMcpToolDefinition`'s handler (`src/integrations/mcp/index.ts:209-237`) discards `extra._meta` (available today in `@modelcontextprotocol/sdk@1.29.0`'s `RequestHandlerExtra`) instead of forwarding it into tool execution, and `FromMCPToolsOptions['callTool']` (lines 78-84) has no `_meta` parameter for outbound propagation. Given armorer already depends on `@opentelemetry/api`, this is the one item in this audit that's actionable now rather than SDK-gated.
2. **`fromMcpTools`'s inbound JSON-Schema-to-Zod conversion has no `$ref`/`$defs` support and fails permissively rather than loudly.** `src/json-schema-to-zod.ts` silently produces `undefined` for unrecognized keywords, and `fromMcpTools` (`src/integrations/mcp/index.ts:168`) falls back to `z.object({}).passthrough()` for the whole tool when its top-level schema can't be resolved — meaning a `$ref`-based MCP tool schema from a spec-conformant 2020-12 server would silently become an "accept anything" Toolbox tool instead of erroring. This is a correctness/security-adjacent gap independent of the 2026-07-28 timeline: it already applies against today's (2025-11-25) spec, since 2020-12 `$ref` support has been a MUST since at least `2025-06-18`.
3. **`toMcpToolDefinition`'s outbound tool schemas have no `$schema` marker and are shaped as draft-07, not 2020-12**, because the SDK's `toJsonSchemaCompat` call (`node_modules/@modelcontextprotocol/sdk/dist/esm/server/mcp.js:76-81`) omits the `target` option. This is technically an SDK-side default, but armorer could compensate by supplying `MCPToolConfiguration.schema` explicitly with a `$schema: "https://json-schema.org/draft/2020-12/schema"` marker for tools where dialect precision matters — currently no such override path is documented or exercised in `documentation/mcp.md`.
4. **No `CreateMCPOptions` hook exists for a caller to supply `ttlMs`/`cacheScope` even in anticipation of an SDK upgrade.** When `@modelcontextprotocol/sdk` eventually ships 2026-07-28 support, closing the "TTL caching" gap will require both an SDK bump _and_ new configuration surface on `createMCP`/`MCPToolConfiguration` — worth designing that option shape ahead of time rather than as a rushed follow-on to the SDK bump.
